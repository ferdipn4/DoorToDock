In the evening it's the reverse: I want to grab a bike and go, but the nearest stations are empty.

Limitations & Trade-offs

Data limitations

24 days of training data. Limits weather diversity, seasonal pattern detection, and ML model performance. The forecast model is particularly bottlenecked - more data would likely improve tree-based models over the baseline.
Minimal heavy rainfall in the collection period. We observed rain on a handful of days, but no sustained storms or snow. Weather features in the model are undertrained.
Sensor active for 16 of 24 days. The KY-028 temperature sensor validated the API data source and was retired - the model uses API temperature for production reliability.
Forecast accuracy is approximate (-- docks MAE). The system communicates this uncertainty through confidence indicators rather than hiding it.

Architecture decisions

Each technical choice involved a trade-off. Here's what we chose and why.

Polling frequency
1/min (not 5/min)
Higher resolution captures station transitions that 5-min polling would miss. Storage cost is negligible on PostgreSQL.
Model architecture
Two models (not one)
Nowcast needs live dock state; forecast can't have it. Separate models let each family use optimal features.
Weather API
Open-Meteo (not OpenWeatherMap)
No API key required, unlimited requests. Sensor validation confirmed data quality matches our needs.
Database
Supabase PostgreSQL (not MongoDB)
SQL joins essential for time-series analysis. GROUP BY hour/weekday aggregations power the dashboard and model training.
Hosting
Heroku (not Raspberry Pi)
24/7 uptime with student credits. Worker dyno collects data, web dyno serves the dashboard - independent scaling.
Temperature source
API (sensor retired after validation)
KY-028 confirmed API reliability (r = 0.95). API temperature is more stable for production.
