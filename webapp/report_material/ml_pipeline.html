Why two models?

The lag feature (empty_docks_lag1) captures the most recent dock count and is the single most important predictor. It is available only in real-time when the system can query the TfL API for the current state. For future planning (e.g., tomorrow at 08:30), no live dock count exists, so the model must rely solely on temporal and weather features — resulting in higher error but still useful guidance.

The Historical Average Baseline outperformed Random Forest and Gradient Boosting when the lag feature was removed, because only ~16 days of training data were available. With limited weather diversity, ML models could not learn robust weather-dock relationships. The simpler baseline generalised better.
